# 旅行記編集フォーマット — 「誰と行った旅か」を記憶の中心に置く

この文書は、tojimasaya.com の旅行記を Codex、Claude Code、その他のコーディングエージェントに編集してもらうときの共通プロンプトです。

対象の旅行記と正しい同行者情報を依頼文に添え、このファイル全体を指示として渡してください。

---

## 1. 編集の目的

旅行記を単なる観光地・写真・日程の記録ではなく、**一緒に旅した人たちが何度でも戻ってこられる「共有の思い出」**として編集してください。

読者がページを開いて、できれば最初の5秒以内に次のことを理解できる状態を目指します。

- 誰と行った旅なのか
- いつ、どこへ行ったのか
- その人たちに何が起きた旅なのか
- どこから順番に読めばよいのか
- 同じ人たちと行った、ほかの旅はあるか

同行者は補足情報ではありません。行き先、日付、写真枚数と同じか、それ以上に大切な旅行記の基本情報です。

---

## 2. 最初に確認する情報

編集前に、対象ページ、本文、写真のキャプション、既存の旅一覧を確認してください。そのうえで、次の情報を確定します。

- 旅行名
- 年月または期間
- 同行者
- 同行者との関係（妻、息子、家族、友人、仕事仲間など）
- 訪問地と読む順番
- その旅を象徴する出来事
- 同じ同行者と行った別の旅行
- 同行者が写っている写真の有無

### 推測禁止

同行者は、本文や写真から推測して確定してはいけません。

- 情報が明確なら、その事実を使用する
- 複数回の旅行で同行者が異なる場合は、旅行ごとに分ける
- 情報が曖昧なら、実装前にユーザーへ確認する
- 写真に写っている人物を、根拠なく特定しない
- 「以前も同じ人と行ったはず」など、会話や本文にない関係を作らない

例：台湾4回の旅行は一括して「夫婦旅」としない。2024年と、2025年の2回は「妻との二人旅」、2026年は「友人たちとの旅」と、訪問ごとに正しく表示する。

---

## 3. 必須のページ構成

### A. 旅行記トップのヒーロー

ヒーロー部分には、旅行名だけでなく同行者が分かる短いサブタイトルを入れます。

良い例：

- `YKK師匠、バスターさん、TJ。三人のカメラ旅はライツパークから`
- `YKK師匠、バスターさん、TJ。いつもの三人でミラノからローマへ`
- `妻との三つの旅、友人たちとの一つの旅`
- `師匠と二人、広州から54時間の列車でラサへ`

避ける例：

- `忘れられない旅`
- `美しい景色を巡る`
- `仲間との旅行`

抽象的な表現だけでは、誰の旅か伝わりません。

### B. 基本情報のピル

ルート、年月、日数、都市数、写真枚数などと並べて同行者を表示します。

表記例：

- `妻との二人旅`
- `TJ・師匠の二人旅`
- `YKK師匠・バスターさん・TJ`
- `友人たちとの旅`
- `妻との二人旅 3回 / 友人との旅 1回`

複数回の訪問をまとめたページでは、同行者の違いが分かるように回数または訪問年を明示します。

### C. 「この旅の仲間」セクション

まとまった一つの旅行には、ヒーロー直後の早い位置に `trip-context` を使った同行者紹介を置きます。

必須要素：

- `この旅の仲間` などの短い見出し
- 同行者が分かるタイトル
- 誰が何を目的に参加したかが伝わる2〜4文
- 旅の特徴を示す短いファクト
- 最初の章へのリンク
- 旅行全体を共有するボタン
- 可能なら同行者が写った写真

文章は人物紹介の羅列ではなく、旅の関係性を表現してください。

例：

> ライカ一筋のYKK師匠、車好きのバスターさん、そしてカメラを手にしたTJ。三人それぞれの「行きたい」をつなぎ、ドイツを走ってプラハまで向かった旅です。

### D. 訪問地・章の入口

旅行記トップには、各都市や各日の章へ移動できる入口を置きます。既存の `journey-overview`、`journey-acts`、`journey-act-link` を優先して再利用してください。

章タイトルは地名だけではなく、その場所で同行者に起きた出来事を表します。

良い例：

- `聖地で最初の珍事件`
- `消えたベンツ、笑うフォトキナ`
- `三人で挑んだ2キロ`
- `最後の最後に大遅刻`
- `旅の後から届いた一枚`

地名、出来事、短い補足の三段階で読める構成が理想です。

### E. 各詳細ページのシリーズ案内

各都市・各日ページのヒーローと本文の間に、`trip-series-context` を使ったシリーズ案内を置きます。

必須要素：

- 同行者
- `1 / 4`、`3 / 6` など全体の中の位置
- 旅行記トップへのリンク
- 旅行全体を一言で説明する文
- `全6都市の目次 →` などの明確な戻り口

複数回訪問をまとめた旅行記では、各回の同行者をここでも明示します。

例：

- `妻との二人旅 — 1 / 4`
- `YKK師匠・バスターさん・TJ — 3 / 6`
- `台湾の旅 — 4 / 4 / 友人たちとの旅`

### F. 旅の一覧ページ

`trips.html` の旅行カードにも同行者を表示します。

- メタ情報に同行者のピルを追加する
- 紹介文の最初の1〜2文で同行者を登場させる
- 行き先の説明だけで終わらせない
- 旅行記トップへのリンクは維持する

一覧だけを見ても、「これは誰と行った旅か」が分からなければいけません。

### G. 同じ人たちとの別の旅

同じ同行者との旅行が複数ある場合、旅行記トップの後半に `related-journeys` を使って相互リンクします。

見出し例：

- `この三人で行った、ほかの旅`
- `妻と歩いた、ほかの場所`
- `家族で残した、別の記憶`

関連性は同行者を基準にします。単に国が近い、年代が近いという理由だけで並べないでください。

### H. 共有の出口

旅行記トップの最後に `closing-share` を置きます。

- 一緒に旅した人へ送りやすい文章にする
- `js/trip-share.js` を再利用する
- `<main>` に `data-trip-share-title` と `data-trip-share-text` を設定する
- 外部の新しい共有ライブラリは追加しない

---

## 4. 写真の選び方

旅行記トップの同行者紹介では、景色だけの写真より、同行者が写っている写真を優先します。

優先順位：

1. 全員または主要な同行者が自然に写っている写真
2. その旅らしい食卓、移動中、出来事の写真
3. 人物写真がない場合は、旅を象徴する場所の写真

注意事項：

- 写真の人物名は確実な場合だけ書く
- `alt` には、分かる範囲で場所、状況、同行者を記述する
- 本人の許可が確認できないフルネームを新たに公開しない
- 既存画像を優先し、不要な画像生成や画像加工をしない
- 写真の縦横比がカードに合わない場合も、人物の顔が不自然に切れないよう確認する

---

## 5. 文章の編集方針

### 人物と出来事を先に書く

観光情報より先に、誰と何が起きたかを書きます。

`フィレンツェを訪問した。` よりも、`三人で2キロのビステッカに挑んだフィレンツェ。` の方が、この旅行記に適しています。

### 本文にある具体的な記憶を使う

次の要素を優先して拾います。

- 失敗や勘違い
- 食卓での出来事
- 同行者の希望で立ち寄った場所
- 後から笑い話になった事件
- 写真に残らなかったが、本人がよく覚えている瞬間
- その人らしさが表れている行動や会話

### 語り口

- ユーザーの既存の一人称とユーモアを残す
- 日本語で自然に書く
- 過度に感動的、広告的、詩的にしない
- 本文にない気持ちや会話を創作しない
- 既存本文を必要以上に書き換えない
- `妻`、`師匠`、`バスターさん` など、本人が使っている呼び方を尊重する

---

## 6. 正確性とプライバシー

- 同行者情報はユーザーが示した事実を最優先する
- 本文、写真、メタデータ、一覧ページの表記を一致させる
- 同じ旅行でも途中参加・現地合流の場合は、誤解のない表現にする
- 名前が公開済みでも、フルネームの追加が必要か慎重に判断する
- 家族の個人情報、顔、年齢、学校、住所などを新たに推測・追加しない
- 同行者写真がなくても、別人の写真を代用しない
- 誤りを見つけた場合は、見た目の改善より事実の修正を優先する

2026年の台湾旅のように、宿泊は一人でも現地で友人と過ごすことが旅の中心なら、本文に合わせて `友人たちとの旅` と表現できます。ただし、全行程を常に一緒に移動したような書き方は避けます。

---

## 7. メタデータと検索導線

旅行記トップと詳細ページでは、次を確認・更新します。

- `<title>`
- `meta description`
- canonical URL
- Open Graph title / description / image
- Twitter Card title / image
- JSON-LD の headline / description / image / dateModified
- 旅の一覧からの内部リンク
- 同じ同行者との関連旅行リンク

description には、可能なら同行者、行き先、年月、代表的な出来事を自然な文章で含めます。

例：

`2016年9月、YKK師匠・バスターさん・TJの三人でミラノからローマまで6都市を縦断した旅行記。`

検索キーワードを不自然に詰め込まず、初めて見る人にも内容が伝わる説明を優先してください。

---

## 8. 実装上のルール

このサイトは plain HTML、CSS、小さなJavaScriptで構成されています。

- 新しいビルドツールやフレームワークを導入しない
- 小さく、元に戻しやすい変更にする
- 既存の写真と `assets/` 以下のファイルを維持する
- 既存の `data/` ファイルを壊さない
- 共通部品は `style.css` と `js/trip-share.js` を再利用する
- 既存の `trip-context`、`journey-overview`、`trip-series-context`、`related-journeys`、`closing-share` を優先する
- ページ固有の重複CSSを増やさない
- 新しい外部トラッキングスクリプトを追加しない
- note、DRONE.jp、X、Facebook、YouTube、メールの既存リンクを勝手に削除しない
- 新しい公開ページを作った場合だけ、既存ページからリンクし `sitemap.xml` に追加する
- 既存ページの編集だけなら、不要なサイトマップ変更を行わない
- 作業前に `git status` を確認し、関係のない変更を上書きしない

## 9. 完了前の確認

### 内容

- [ ] 旅行記トップを見て、誰と行った旅かすぐ分かる
- [ ] 複数回の訪問で同行者が異なる場合、回ごとに区別されている
- [ ] 詳細ページにも同行者とシリーズ内の位置がある
- [ ] 旅一覧にも同行者が表示されている
- [ ] 同じ同行者との別の旅へ移動できる
- [ ] 本文にない人物関係や出来事を創作していない

### 技術

- [ ] `git diff --check` が成功する
- [ ] 変更したインラインJavaScriptとJSON-LDが構文エラーにならない
- [ ] 変更ページのローカル `href`、`src`、`srcset` が存在する
- [ ] canonical、OG、JSON-LDがページ内容と一致する
- [ ] PC表示でレイアウトが崩れない
- [ ] スマートフォン表示で横方向にはみ出さない
- [ ] 人物写真の顔や本文が不自然に切れない
- [ ] ブラウザのコンソールに新しいエラーがない
- [ ] 共有ボタンがある場合、既存の共有処理が読み込まれている

## 10. 作業完了時の報告

報告は、変更したファイルの列挙だけで終わらせないでください。

次の順番で簡潔に伝えます。

1. この旅行記を、誰と行ったどんな旅として再編集したか
2. 旅行記トップで何が分かるようになったか
3. 詳細ページと旅一覧に何を追加したか
4. PC・スマートフォン、リンク、JavaScriptをどう検証したか
5. 未確認事項や、ユーザーに確認が必要な同行者情報が残っているか

## 11. 依頼時に添えるテンプレート

以下をコピーし、空欄を埋めてこのファイルと一緒に渡してください。

```text
このリポジトリの旅行記を、TRAVEL_JOURNAL_FORMAT.md のルールに従って編集してください。

対象の旅行記：
対象ページ：
旅行年月：
同行者：
同行者との関係：
旅行の順番・章数：
同じ同行者との関連旅行：
特に残したい出来事：
使用してよい同行者写真：
公開したくない名前・情報：

既存の文章と写真を尊重し、同行者情報を推測せず、実装後にPCとスマートフォンで確認してください。
```

情報が一部空欄でも、リポジトリ内で明確に確認できる事実は使用できます。ただし、同行者、人物名、関係性が曖昧な場合は必ずユーザーへ確認してください。
